# 3. 重要な実装課題

## 3.1 課題1：モンゴメリ簡約

**問題**: 初期実装がQINV定数の誤りにより失敗。

**症状**:
```
Julia NTT: [-3148987, -11171733, ...]
C NTT:     [287814, -6262710, ...]
```

**根本原因**: 誤ったQINV値の使用（58728449ではなく20088143）

**解決策**:
```julia
const QINV = 58728449

function montgomery_reduce(a::Int64)
    t = Int32(a * QINV)
    t = Int32((a - Int64(t) * Q) >> 32)
    return t
end
```

**教訓**: モンゴメリ演算では特に、参照実装に対して常に定数を検証すること。

**費やした時間**: デバッグに約2時間

---

## 3.2 課題2：配列インデックス（0ベースvs 1ベース）

**問題**: C言語（0インデックス）とJulia（1インデックス）間でNTTループインデックスが不整合。

**C言語コード**:
```c
k = 0;
for(len = 128; len > 0; len >>= 1) {
    zeta = zetas[++k];  // zetas[1]から開始
}
```

**修正されたJulia**:
```julia
k = 0
for len in [128, 64, 32, 16, 8, 4, 2, 1]
    k += 1
    zeta = ZETAS[k + 1]  # +1が必須
end
```

**教訓**: オフバイワンエラーは最も一般的なバグ。Julia配列アクセスには常に+1を追加。

**費やした時間**: デバッグに約3時間

\newpage

## 3.3 課題3：整数オーバーフローのセマンティクス

**問題**: JuliaとC言語でInt32オーバーフローの扱いが異なる。

**解決策**: 明示的なモジュラー演算を使用。

**費やした時間**: 約1時間

---

## 3.4 課題4：メモリレイアウトとパッキング

**問題**: SubArrayビューがパッキング関数で黙って失敗。

**結果**: パッキング後、公開鍵t1が全てゼロに！

**解決策**: 中間バッファパターン
```julia
function pack_pk(pk::Vector{UInt8}, rho::Vector{UInt8}, 
                 t1::PolyVec)
    for i in 0:(K-1)
        tmp = zeros(UInt8, POLYT1_PACKEDBYTES)
        polyt1_pack(tmp, t1.polys[i+1])
        pk[offset+1:offset+POLYT1_PACKEDBYTES] = tmp
    end
end
```

**教訓**: Juliaの型システムはCポインタより厳格。一時バッファを使用する。

**費やした時間**: 約4時間（最もデバッグが困難）

---

## 3.5 課題5：パラメータの不整合（CRHBYTES）

**問題**: 誤った定数が秘密鍵サイズの誤りを引き起こす。

**症状**:
```
Julia: CRYPTO_SECRETKEYBYTES = 2560
C:     CRYPTO_SECRETKEYBYTES = 2544
差分: 16バイト
```

**根本原因**: CRHBYTES = 64（誤り） → 48（正しい）

**費やした時間**: 約2時間

\newpage

## 3.6 課題6：ステートフルSHAKE vs ステートレスAPI

**問題**: C言語はステートフルなKeccakコンテキストを使用。

**解決策**: ステートフル動作をエミュレート
```julia
function poly_uniform(a, seed, nonce)
    buf = shake128(buflen + 3, input)
    ctr = rej_uniform(a.coeffs, 0, N, buf, buflen)
    
    while ctr < N  # 段階的なスクイーズ
        new_buf = shake128(...)
        ctr += rej_uniform(...)
    end
end
```

**教訓**: ステートフルな暗号APIは関数型言語で慎重にエミュレート。

**費やした時間**: 約3時間

# 6. 得られた教訓

## 6.1 一般原則

1. **全ての定数を検証**: 仮定せず、参照を確認
2. **早期にCに対してテスト**: バグが複雑化する前に捕捉
3. **インデックスマッピングは重要**: 全ての配列アクセスに+1
4. **型の厳格性が役立つ**: コンパイル時に多くのバグを捕捉
5. **黙った失敗が最悪**: アサーションを自由に追加

## 6.2 Julia固有の知見

**長所**:

- 型システムが多くのエラーを捕捉
- REPLテストが開発を加速
- モジュールシステムがコードを整理
- FFI（ccall）が簡単

**短所**:

- SubArray vs Vectorの混同
- 整数オーバーフローのセマンティクスがC言語と異なる
- 暗黙的なポインタ演算がない
- ステートフルAPIにはエミュレーションが必要

## 6.3 暗号固有

1. **モンゴメリ演算は微妙**: まずQINVを正しく設定
2. **棄却サンプリングには注意**: カウンタのオフバイワンが黙って失敗
3. **定数時間は困難**: Juliaは保証しない
4. **パッキングはエラーが起きやすい**: 広範なテストが必要
